# Contributing to Celstomp Documentation

Thank you for your interest in contributing to the Celstomp documentation! This guide will help you get started.

## Ways to Contribute

### Reporting Issues

Found a typo, error, or unclear explanation? Please open an issue:

1. Go to the [Issues page](https://github.com/ginyoa/celstomp_v1/issues)
2. Click "New Issue"
3. Use the "Documentation" label
4. Describe the problem clearly

### Suggesting Improvements

Have ideas for new content or improvements?

- Missing topics
- Better examples
- Clarifications
- New guides

Open an issue to discuss your ideas before writing.

### Writing Content

Want to write or edit documentation?

1. Fork the repository
2. Create a new branch
3. Make your changes
4. Submit a pull request

## Development Setup

### Prerequisites

- [mdBook](https://rust-lang.github.io/mdBook/guide/installation.html) installed
- Git
- A text editor (VS Code recommended)

### Running Locally

```bash
# Clone your fork
git clone https://github.com/YOUR_USERNAME/celstomp_v1.git
cd celstomp_mdbook

# Serve locally
mdbook serve

# Open http://localhost:3000
```

## Writing Guidelines

### Style

- **Clear and concise** - Get to the point
- **Active voice** - "Click the button" not "The button should be clicked"
- **Present tense** - "The tool saves" not "The tool will save"
- **Second person** - "You can" not "Users can"

### Formatting

- Use Markdown for all content
- Code blocks should specify language
- Tables for structured data
- Lists for steps or options

### Structure

Each page should have:

1. **Title** (# H1 heading)
2. **Introduction** - What this page covers
3. **Main content** - Organized with H2/H3
4. **Examples** - Where helpful
5. **See Also** - Links to related pages

### Code Examples

All code should be:

- **Tested** - Actually works
- **Complete** - Can be copy-pasted
- **Explained** - Comments for complex parts
- **Realistic** - Practical examples

### Screenshots

When adding screenshots:

- Use consistent browser/theme
- Crop to relevant area
- Annotate if helpful
- Optimize file size

## Review Process

1. **Submit PR** with clear description
2. **Maintainer review** within a few days
3. **Address feedback** if requested
4. **Merge** when approved

## Content Types

### User Guides

Step-by-step instructions for users:
- Clear headings
- Numbered steps
- Screenshots
- Tips and warnings

### Reference

Technical documentation:
- Complete information
- Structured data (tables)
- Code examples
- Accuracy is critical

### Concepts

Explanations of how things work:
- Theory and context
- Diagrams if helpful
- Examples
- Links to guides

## Questions?

- Open an issue for discussion
- Comment on existing issues
- Reach out to maintainers
